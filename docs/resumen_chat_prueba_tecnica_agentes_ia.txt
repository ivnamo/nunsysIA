1. ALCANCE DE LA POC
====================

La POC implementa un sistema agentic empresarial capaz de responder preguntas
de negocio en lenguaje natural combinando:

- ERP Northwind reducido con SQLite en memoria.
- API REST mock de produccion.
- Documentos PDF indexados mediante RAG.
- API principal FastAPI.
- UI conversacional Chainlit.
- Trazabilidad publica de fuentes, pasos, tool calls, fallbacks y estado.

La arquitectura acordada es:

- FastAPI para backend.
- LangGraph para orquestacion.
- LangChain para tools, LLM, embeddings y RAG.
- Chainlit para demo conversacional.
- ChromaDB como vector store objetivo, con fallback en memoria.
- Pydantic para contratos.
- pytest para validacion automatizada.

3. CONTRATO DE CONSULTA
=======================

Endpoint principal:

POST /api/query

Entrada:

{
  "question": "Que pedidos pendientes tiene el cliente ALFKI y en que estado de produccion estan?",
  "conversation_id": "demo-001"
}

Salida:

{
  "answer": "...",
  "sources": ["ERP", "Produccion"],
  "reasoning": ["..."],
  "tool_calls": [],
  "fallbacks": [],
  "confidence": 0.9,
  "status": "completed",
  "data": {},
  "failure_reason": null
}

Estados permitidos:

- completed
- partial_answer
- insufficient_context
- tool_error
- failed
- unsupported


4. CASOS DE DEMO CUBIERTOS
==========================

ERP y produccion:

- Pedidos pendientes de ALFKI y estado de produccion.
- Pedidos bloqueados y motivo.
- Clientes con pedidos retrasados por problemas de produccion.
- Resumen de pedidos del mes.

RAG:

- Penalizaciones por retrasos.
- Plazos de entrega.
- Resumen de contrato.
- Preguntas sin evidencia documental con `insufficient_context`.

Mixto:

- Penalizaciones por pedido segun estado ERP/produccion y documentos.

Memoria conversacional:

- Primera consulta: pedidos pendientes de ALFKI.
- Follow-up con el mismo `conversation_id`: `Y en que estado estan?`.
- Conversacion distinta sin historial: `unsupported`.


5. TRAZABILIDAD
===============

Cada respuesta debe exponer:

- fuentes consultadas;
- pasos visibles ejecutados;
- tool calls con argumentos sanitizados;
- fallbacks usados;
- estado final;
- resumen publico de evidencias en `data`.

La respuesta no debe exponer:

- chain-of-thought interno;
- prompts internos;
- secretos;
- connection strings;
- filas raw de ERP;
- respuestas raw de servicios internos;
- textos completos de chunks documentales.


6. DECISIONES VIGENTES
======================

- El ERP runtime actual usa SQLite en memoria y seed `data/northwind_seed.sql`.
- `ERP_DATABASE_URL` queda reservado para persistencia externa futura.
- `DATABASE_URL` no se usa para ERP porque Chainlit la reserva para su data layer.
- ChromaDB es el vector store objetivo. Si no esta disponible, el sistema usa
  fallback en memoria y lo declara en `fallbacks`.
- Gemini y OpenAI son opcionales; los tests no dependen de llamadas pagadas.
- La memoria conversacional no es fuente de verdad de negocio: solo resuelve
  referencias conversacionales y debe combinarse con ERP, Produccion o
  Documentos cuando la respuesta requiera datos actuales.
