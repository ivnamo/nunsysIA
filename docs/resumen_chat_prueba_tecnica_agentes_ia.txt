1. REQUISITOS DEL SISTEMA EXTRAÍDOS DEL PDF
===========================================

1.1. Objetivo general del sistema
---------------------------------

Se debe desarrollar una POC de sistema multi-agente inteligente capaz de responder preguntas de negocio en lenguaje natural, integrando varias fuentes de datos empresariales:

- ERP basado en la base de datos Northwind.
- Sistema de producción accesible mediante API REST.
- Documentos PDF mediante RAG.
- Capacidad de generar respuestas consolidadas, explicables y trazables.

El sistema debe interpretar consultas, planificar acciones, ejecutar llamadas a diferentes fuentes y generar respuestas finales comprensibles.


1.2. Requisitos funcionales obligatorios
----------------------------------------

A) Consultas en lenguaje natural sobre ERP y producción

El sistema debe responder preguntas de negocio como:

- "¿Qué pedidos pendientes tiene el cliente X y en qué estado de producción están?"
- "¿Qué pedidos están bloqueados y cuál es el motivo?"
- "¿Qué clientes tienen pedidos retrasados por problemas de producción?"
- "Dame un resumen del estado de los pedidos de este mes"

Para ello, debe ser capaz de:

- Consultar datos del ERP:
  - Clientes.
  - Pedidos.
  - Importes.
- Consultar el sistema de producción mediante API REST.
- Combinar información de ERP y producción.
- Generar una respuesta consolidada y comprensible.


B) Consultas sobre documentos con RAG

El sistema debe permitir:

- Subir documentos en formato PDF.
- Indexar esos documentos en una base de datos vectorial.
- Hacer preguntas sobre el contenido de los documentos.

Ejemplos de consultas:

- "¿Qué dice este documento sobre plazos de entrega?"
- "Resume los puntos clave del contrato"
- "¿Hay alguna penalización por retrasos?"


C) API de consulta

El sistema debe exponer al menos este endpoint:

POST /api/query

Este endpoint debe:

- Recibir una pregunta en lenguaje natural.
- Procesarla mediante el sistema multi-agente.
- Devolver una respuesta estructurada.

Formato esperado de respuesta:

{
  "answer": "El cliente ALFKI tiene 3 pedidos pendientes...",
  "sources": ["ERP", "Producción"],
  "reasoning": [
    "Consulta ERP para pedidos",
    "Consulta API de producción",
    "Fusión de resultados"
  ]
}

2. CÓMO PROBARLO EN UNA POC
===========================

Para una POC no necesitas conectarte a un ERP real desde el primer día. Lo más razonable es simular el ecosistema empresarial completo, pero con integraciones realistas:

- Una base de datos tipo ERP.
- Una API REST de producción.
- Un módulo RAG para PDFs.
- Un agente que orqueste todo.

La clave es que el agente no se invente los datos, sino que use herramientas reales aunque los datos sean de prueba.

3. CÓMO PROBAR LA INTEGRACIÓN CON ERP Y PRODUCCIÓN
==================================================

3.1. Simular el ERP con Northwind
---------------------------------

Como el PDF menciona Northwind, puedes usar una base de datos local con ese esquema.

Opciones:

- PostgreSQL.
- SQLite.
- SQL Server.
- MySQL.

Para la POC, se recomienda PostgreSQL en Docker porque queda más profesional y se parece más a un entorno real.

Tablas mínimas útiles de Northwind:

- customers
- orders
- order_details
- products
- employees
- shippers

Para las preguntas del PDF necesitas especialmente:

- customers
- orders
- order_details

Ejemplo de datos ERP:

Customer ALFKI
  Order 10248 - pendiente - importe 440 €
  Order 10252 - pendiente - importe 1863 €
  Order 10255 - enviado - importe 2490 €

El agente debería tener una tool tipo:

def get_customer_pending_orders(customer_id: str):
    """
    Consulta en el ERP los pedidos pendientes de un cliente.
    """

Internamente esa tool ejecutaría SQL contra Northwind.


3.2. Simular el sistema de producción con una API REST
------------------------------------------------------

El ERP te dice que existe un pedido, cliente e importe.

Pero el estado productivo puede venir de otro sistema, por ejemplo:

Pedido 10248 -> En fabricación
Pedido 10252 -> Bloqueado por falta de material
Pedido 10255 -> Finalizado

Para probarlo, creas una API REST fake con FastAPI.

Endpoints mínimos:

GET /production/orders/{order_id}
GET /production/orders?status=blocked
GET /production/orders/delayed

Ejemplo de respuesta:

{
  "order_id": "10252",
  "production_status": "blocked",
  "blocked_reason": "Falta de material",
  "estimated_finish_date": "2026-03-22"
}


3.3. Flujo de prueba principal
------------------------------

Pregunta:

"¿Qué pedidos pendientes tiene el cliente ALFKI y en qué estado de producción están?"

Flujo esperado:

1. El agente detecta que necesita datos de ERP.
2. Llama a la tool ERP.
3. La tool ERP devuelve los pedidos pendientes del cliente ALFKI.
4. El agente detecta que necesita el estado productivo de esos pedidos.
5. Llama a la API REST de producción por cada pedido.
6. Fusiona los datos.
7. Devuelve una respuesta consolidada.
8. Incluye fuentes y razonamiento.

Respuesta esperada:

{
  "answer": "El cliente ALFKI tiene 2 pedidos pendientes. El pedido 10248 está en fabricación y el pedido 10252 está bloqueado por falta de material.",
  "sources": ["ERP", "Producción"],
  "reasoning": [
    "Se consultó el ERP para localizar los pedidos pendientes del cliente ALFKI.",
    "Se consultó la API de producción para obtener el estado de los pedidos 10248 y 10252.",
    "Se combinaron los datos comerciales del ERP con los estados productivos."
  ]
}


3.4. Pruebas manuales recomendadas
----------------------------------

Prueba 1: consulta directa al ERP

Pregunta:

"¿Qué pedidos pendientes tiene el cliente ALFKI?"

Validas que solo consulte el ERP.

Resultado esperado:

{
  "sources": ["ERP"],
  "reasoning": [
    "Consulta ERP para obtener pedidos pendientes del cliente ALFKI."
  ]
}


Prueba 2: consulta ERP + producción

Pregunta:

"¿Qué pedidos pendientes tiene el cliente ALFKI y en qué estado de producción están?"

Aquí debe consultar:

ERP -> pedidos
Producción API -> estado de cada pedido

Resultado esperado:

{
  "sources": ["ERP", "Producción"]
}


Prueba 3: producción primero y ERP después

Pregunta:

"¿Qué clientes tienen pedidos bloqueados por problemas de producción?"

Orden lógico:

1. Consultar API de producción para encontrar pedidos bloqueados.
2. Extraer los order_id.
3. Consultar ERP para saber a qué clientes pertenecen.
4. Devolver cliente + pedido + motivo de bloqueo.

Este caso es muy bueno para demostrar razonamiento multi-fuente.


Prueba 4: resumen mensual

Pregunta:

"Dame un resumen del estado de los pedidos de este mes."

El sistema debería:

1. Consultar ERP para pedidos del mes.
2. Consultar producción para sus estados.
3. Agrupar por estado.
4. Calcular importes si procede.
5. Devolver resumen.

Ejemplo:

{
  "answer": "Este mes hay 12 pedidos: 5 en fabricación, 3 bloqueados, 2 retrasados y 2 finalizados. El importe total afectado por bloqueos es de 8.420 €.",
  "sources": ["ERP", "Producción"],
  "reasoning": [
    "Se consultaron los pedidos del mes en el ERP.",
    "Se consultó el estado productivo de cada pedido.",
    "Se agruparon los pedidos por estado y se calcularon importes."
  ]
}

def test_query_customer_pending_orders_with_production_status():
    response = client.post("/api/query", json={

CONCLUSIÓN
==========

La POC debe demostrar principalmente que el sistema puede:

1. Recibir preguntas en lenguaje natural.
2. Decidir qué fuentes consultar.
3. Consultar una base ERP tipo Northwind.
4. Consultar una API REST de producción.
5. Consultar documentos mediante RAG.
6. Fusionar datos de varias fuentes.
7. Responder de forma estructurada.
8. Incluir sources, reasoning y trace.
9. Ser ejecutable con Docker.
10. Tener una interfaz gráfica tipo Chainlit.

La parte más importante para la evaluación será que la integración ERP + producción sea real dentro de la POC: una base de datos con datos controlados, una API separada y una respuesta final que demuestre qué se consultó y cómo se fusionaron los datos.
